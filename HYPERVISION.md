# HyperVision — 超视觉暴露映射系统

## 设计目标

在复杂 3D 地形（洞穴、迷宫、多层建筑、空岛）中，超越传统 2D ESP 的局限性，提供**可见性驱动的导航与态势感知**。

---

## 核心理念

### 从 "看到位置" 到 "看到看的位置"

```
传统 ESP:      你在箱子里 → 所有敌人标红点 → 但隔墙你不知道怎么打
HyperVision:   你在箱子里 → 标记出"走到哪个位置能看到敌人"
```

不追求导航到敌人脸上，而是导航到 **任何能看到敌人的位置**，让玩家自己选择进攻路线。

### 暴露体积 (Visibility Volume)

对每个目标的 **全环绕 3D 可见性采样**：

```
采样算法:
  在目标周围生成球形采样点（步长可配置，默认 50-100 单位）
  层高遍历: 每 50 单位高度一层，覆盖目标 Z ±1000 单位
  对每个采样点 → 引擎 LineTrace 到目标
      如果无遮挡 → 标记为暴露点
      如果有遮挡 → 丢弃（除非目标本身被标记为"允许部分遮挡"）
  暴露点集构成目标的暴露体积
```

暴露体积的形状**完全自适应**：
- 门缝 → 薄扇形
- 窗户 → 梯形棱柱
- 开阔地 → 半球壳
- 复杂洞穴 → 洞穴内腔形状
- 多层结构 → 每层独立的暴露环

### 贯通线

传统 2D 雷达把洞穴顶部误判为墙。3D 采样自然解决：

```
俯视图以为:  ████████ 墙
实际剖面:    顶部岩石
            洞穴内部 ← 贯通的空间
            底部岩石
LineTrace 从外部进入洞穴→ 直通 → 暴露有效
```

贯通线即 **LineTrace 穿透洞穴开口的自然结果**，不需要特殊处理。

## 地形扫描与雷达绘制

暴露体积解决"能看到敌人吗"，地形解决"地图长什么样"。两者叠加才是完整的态势感知。

### 线型编码体系

传统雷达只切一个 Z 平面绘制墙线，在洞穴悬挑、多层建筑中完全失效。改用拓扑线型区分垂直层次：

```
Z+2:  ····楼板····       淡灰点线     → 上方两层
Z+1:  - - -横梁- - -     中灰虚线     → 上一层 
当前: ━━━━墙壁━━━━        白色实线     → 玩家所在高度
Z-1:  ░░░阶梯░░░         棕色虚线     → 下一层
Z-2:  ═══地基═══         深色粗线     → 底部结构

特殊地形标注:
-•-•- 悬挑线 → 头顶有岩石/楼板，脚下是空的（洞穴、阳台）
~ ~ ~  波浪线 → 斜坡/楼梯（高度过渡区域）
[▓▓▓]  实心块 → 完全不可通过的障碍物
◉      节点   → 洞口/门/窗口（垂直贯通位置标记）
```

### 双图层雷达

```
图层1: 地形骨架（永久）
  ━━━━━━━━━━━━━━━━━━━━━━━━
  ║  [◉]               ║
  ║    ━━━[◉]━━━       ║
  ║                    ║
  ━━━━━━━━━━━━━━━━━━━━━━━━
  显示: 墙壁、门洞、楼梯、悬挑结构
  来源: bridge DLL scan_terrain
  更新: 地图切换时刷新，或手动刷新

图层2: 暴露映射（动态叠加）
  ░░░░░░░░░░
  ░░[P]━━░▔▔[E]░░
  ░░░░░░░░░░
  显示: 暴露体积 + 导航路径
  来源: HyperVision visibility_scan
  更新: 实时（2-5 Hz）
```

### scan_terrain 命令

```
输入: center_pos, range=(5000, 5000), z_samples=5, z_range=1000
过程: 
  1. 遍历 UWorld → PersistentLevel → Actors
  2. 对每个 UStaticMeshComponent:
     a. 读 Bounds (origin, box_extent)
     b. 用 ComponentToWorld 变换到世界坐标
     c. 对每个 Z 切片层计算包围盒与切面的交线段
     d. 分类: 悬挑/实心/斜坡/洞口
  3. 合并共线线段（Douglas–Peucker 简化）
输出: 每层的 [(x1,y1,x2,y2, type, z)], type∈{wall,overhang,ramp,door,open}

性能:
  - 只在换图时跑一次（约 5-15ms）
  - 缓存直到检测到 Level 变化
  - 线简化后通常 <500 条线段
```

### 桥接层：暴露映射引擎 (MecchaESP_HV)

由 bridge DLL 实现的独立模块，通过 TCP 协议暴露命令。

#### TCP 命令集

```
visibility_scan
  输入: target_pos, sampling_step=80, z_layers=20
  输出: exposure_cloud（暴露点点云, 3D坐标列表）
         stats（采样数、命中率、层级分布）

path_find
  输入: player_pos, target_pos, exposure_cloud
  输出: paths（路径列表, 每条为3D坐标序列, 按距离排序）
         occluded_zones（被遮挡的区域体积数据）

start_hypervision
  输入: target_list, mode="auto"/"manual", path_count=3
  效果: 启动持续采样+绘制环路（运行在bridge工作线程）

stop_hypervision
  效果: 停止采样, 清除所有debug绘制
```

## 3D 视觉表达（引擎内绘制）

全部经由 bridge DLL 调 `DrawDebug*` 函数，仅本机可见。

### 路径线

```
视觉:
  ●────────●────────●────────●  
   |        |        |        |
  节点球   线段     节点球   线段

  - 颜色: 最近路径=亮绿(0,255,0)
          次近路径=青色(0,200,200)
          第三路径=黄色(200,200,0)
  - 节点球: DrawDebugSphere, 半径15单位, 半透明
  - 线段: DrawDebugLine, 粗1单位
  - 动态闪烁: 路径末端节点轻微脉动（alpha 0.3-1.0 循环）
```

### 暴露体积

```
暴露点聚类后:
  - 暴露点簇: DrawDebugSphere, 半径8单位, 
              绿色半透明(alpha 0.15)
  - 暴露簇合并为体积: 用 DrawDebugCapsule 或 
              多个 DrawDebugSphere 叠加形成云效果
  - 从玩家到最近暴露点的路径: 粗亮绿线+箭头
```

### 遮挡反馈

```
被遮挡的区域:
  - DrawDebugBox 虚框, 红色半透明
  - 仅在 "debug_mode" 开启时显示
  - 正常模式下只显示可到达的暴露区域
```

## 性能策略

| 场景 | 采样密度 | 频率 | 预期开销 |
|------|---------|------|---------|
| 开阔地 | 低 (100单位步长) | 2 Hz | <0.1ms |
| 建筑内 | 中 (60单位步长) | 3 Hz | 0.3ms |
| 复杂洞穴 | 高 (40单位步长) | 5 Hz | 0.8ms |
| 多目标(4人) | 按距离分配 | 共享10Hz总量 | ≤2ms |

### 优化手段

1. **空间哈希网格** — 把世界分成格子，同一格内的暴露点复用 LineTrace 结果
2. **帧分片** — 每帧只处理 1/4 的采样点，4 帧完成一轮全采样
3. **距离衰减** — 超过 200m 的目标降低采样密度
4. **可见性缓存** — 暴露点在 500ms 内不变则跳过 LineTrace
5. **工作线程** — 采样和 A* 搜索跑在 DLL 工作线程，零阻塞

## 总结

HyperVision 不是更好看的 ESP。它是**思维方式的转变**：

```
之前: "敌人在哪？"  →  一个红点
之后: "在哪能看到敌人？" → 一条路径 + 一片区域
```

雷达不再画墙壁轮廓（那永远画不准），而是画"眼睛能看到的地方"。这个信息是 LineTrace 直接测出来的，不需要理解地图结构、不需要 NavMesh、不需要解析 StaticMesh。**它的精度只取决于采样密度，和地图复杂度无关。**

这也意味着它对空岛、多层建筑、洞穴、迷宫、动态破坏场景全部一视同仁——只要 LineTrace 能穿过的就是通的，穿不过的就是挡的，自适应。
